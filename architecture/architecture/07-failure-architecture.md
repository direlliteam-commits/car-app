# Failure Architecture

Система спроектирована с учётом того, что сбои неизбежны: network failures, realtime interruptions, partial operations, concurrent actions и человеческие ошибки.

Главная цель — при любой ошибке система сохраняет консистентность marketplace и никогда не попадает в логически повреждённое состояние.

## Принципы

### 1. Safe Failure Principle
Любая ошибка должна приводить систему в безопасное состояние, а не в неопределённое.

Если операция не завершена полностью — считается, что она НЕ произошла. Частично применённые изменения запрещены.

### 2. Atomic Operation Rule
Все критические операции выполняются атомарно:
- Subscriptions
- Wallet changes
- Listing visibility
- Ownership changes
- Admin moderation

Система не должна оставлять промежуточных состояний даже при сбое.

### 3. Server Truth Recovery
После любого сбоя клиент обязан восстановить корректное состояние из server truth.

Client state считается временным и восстанавливаемым. Refresh или reconnect всегда возвращает систему к консистентному состоянию.

### 4. Realtime Failure Tolerance
SSE может:
- Задерживаться
- Теряться
- Приходить повторно
- Приходить вне порядка

Realtime архитектура обязана:
- Быть идемпотентной
- Использовать versioning
- Корректно ресинхронизироваться после reconnect

Realtime никогда не является источником истины.

### 5. Retry Safety Rule
Повторное выполнение одной операции не должно менять результат системы.

Любая операция должна быть безопасной при повторном вызове (idempotent behavior).

### 6. Admin Failure Containment
Если admin-действие прерывается:
- Система либо полностью применяет изменение
- Либо полностью откатывает его

Запрещены частичные admin effects.

### 7. Visibility Protection
Listing никогда не должен попасть в состояние:
- Одновременно видим и заблокирован
- Доступен без валидного lifecycle state
- Отображается без permission validation

При ошибке visibility всегда переходит в более безопасное состояние (скрыт, а не опубликован).

### 8. Subscription Consistency Protection
Ошибки оплаты, обновления или изменения подписки не могут:
- Давать лишние права
- Удалять активные права без подтверждения состояния

При неопределённости система выбирает состояние с минимальными привилегиями.

### 9. Data Corruption Prevention
Если данные вызывают конфликт или не проходят validation: они не применяются к системе.

Лучше отклонить изменение, чем сохранить повреждённое состояние.

### 10. Graceful Degradation
При отказе отдельных компонентов:
- Marketplace продолжает работать
- Core функции остаются доступными
- Отключается только вторичная функциональность

Core listings и search всегда имеют приоритет.

### 11. Event Ordering Protection
Система должна корректно обрабатывать события, пришедшие:
- Позже ожидаемого
- Дважды
- После reconnect

Versioning определяет актуальность состояния.

### 12. Failure Transparency
Ошибки не должны создавать скрытые состояния. Любой сбой должен быть:
- Наблюдаемым системой
- Объяснимым через состояние домена
- Восстановимым без ручного вмешательства

### 13. Consistency Over Availability
Если возникает выбор: показать данные быстро ИЛИ показать данные правильно — приоритет всегда отдаётся консистентности marketplace.

Главный принцип: система может временно не работать идеально, но она никогда не должна становиться неправильной.
