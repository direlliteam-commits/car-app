# System Invariants

Фундаментальные законы системы, которые должны оставаться истинными **ВСЕГДА**, независимо от новых функций, изменений UI, admin-действий или realtime событий.

Если любое изменение нарушает invariant — изменение считается архитектурно неправильным.

**Система должна физически не мочь попасть в запрещённое состояние.**

## Инварианты

### 1. Single Truth Invariant
Каждая сущность имеет один источник истины. Ни при каких условиях не допускается существование двух актуальных состояний одной сущности.

Если возникает возможность расхождения — архитектура нарушена.

### 2. Listing Visibility Invariant
Если listing видим в search — он обязан быть доступен при открытии.

Невозможны состояния:
- Найдено в фильтрах, но недоступно
- Отображается в realtime, но отсутствует в server truth

### 3. Ownership Invariant
Каждое объявление всегда имеет:
- Одного владельца
- Одно состояние управления
- Отслеживаемую историю изменений

Listing без владельца или с конфликтующим ownership невозможен.

### 4. Lifecycle Invariant
Listing всегда находится ровно в одном lifecycle state.

Запрещены:
- Частичные состояния
- Смешанные статусы
- Логически необъяснимые переходы

Все переходы состояния должны быть детерминированными.

### 5. Permission Invariant
Ни одно действие не может быть выполнено без серверной проверки прав.

Если действие возможно через обход UI — система считается сломанной.

### 6. Realtime Consistency Invariant
Realtime события никогда не создают новое состояние.

Client state после SSE всегда должен сходиться с server state. После reconnect система обязана восстановить корректное состояние без ручных действий пользователя.

### 7. Subscription Integrity Invariant
Подписки и лимиты не могут находиться в промежуточных состояниях.

Невозможны ситуации:
- Активная подписка без прав
- Права без активной подписки
- Частично применённые лимиты

Все изменения подписок атомарны.

### 8. Admin Determinism Invariant
Любое admin-действие приводит к одному и тому же результату независимо от клиента или времени выполнения.

Admin не может создать скрытое состояние системы.

### 9. Data Normalization Invariant
Одна бизнес-сущность = одна запись истины.

Запрещены:
- Дубли данных
- Денормализованные источники бизнес-логики
- Вычисляемые данные, сохранённые как независимое состояние

### 10. Marketplace Reality Invariant
Marketplace всегда отражает реальное состояние системы.

Невозможны ситуации:
- Пользователь видит устаревшие данные как актуальные
- Разные пользователи видят противоречащие состояния одной сущности

### 11. State Recovery Invariant
Система должна уметь восстановить корректное состояние из server truth в любой момент:
- После reconnect
- После refresh
- После сбоя realtime

Client никогда не является окончательным источником состояния.

### 12. Evolution Safety Invariant
Ни одна новая функция не может нарушать существующие invariants.

Если новая логика требует нарушения invariant — архитектурное решение считается неправильным и должно быть переработано.

Работай как архитектор платформы с формальными законами системы: не исправляй последствия, предотвращай появление невозможных состояний.
