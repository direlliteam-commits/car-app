# Marketplace Core Laws

Система рассматривается как marketplace engine, а не просто приложение.

Главная цель — сохранить целостность ядра маркетплейса при росте пользователей, объявлений, monetization и admin-инструментов.

## Законы

### 1. Listing Is The Core Entity
Объявление (Listing) — центральная сущность всей платформы. Все домены существуют вокруг него:
- Vehicle data
- User ownership
- Subscriptions
- Visibility
- Promotion
- Moderation
- Realtime updates
- Search & filters

Запрещено создавать альтернативные модели объявлений.

### 2. Visibility Is State, Not UI
Видимость объявления — это серверное состояние, а не логика интерфейса. Visibility определяется только через:
- Moderation status
- Subscription state
- Permissions
- Admin decisions
- Lifecycle rules

UI лишь отображает результат.

### 3. Marketplace Trust Law
Пользователь всегда должен видеть консистентную правду системы. Нельзя допускать ситуации, когда:
- Объявление видно в поиске, но недоступно внутри
- Realtime показывает устаревшее состояние
- Фильтры возвращают несогласованные данные

Если возникает конфликт — серверное состояние является единственной истиной.

### 4. Ownership Integrity
У каждого listing всегда один владелец и один источник управления. Нельзя допускать:
- Частичного владения
- Скрытых override без system trace
- Изменений без lifecycle-события

Все изменения ownership должны быть отслеживаемыми событиями.

### 5. Monetization Separation Law
Монетизация не изменяет бизнес-логику listings. Подписки и продвижения:
- Влияют только на visibility и limits
- Не изменяют структуру данных
- Не создают альтернативные типы объявлений

Платёжная логика расширяет систему, но не переписывает её.

### 6. Lifecycle Determinism
Каждое объявление проходит предсказуемый lifecycle:
draft → active → moderated → visible → archived → deleted.

Нельзя существование "непонятных" промежуточных состояний. Любое состояние должно объясняться бизнес-логикой.

### 7. Search Reality Law
Search и Filters всегда отражают реальное состояние marketplace. Фильтры:
- Работают только от нормализованных данных
- Не содержат собственной бизнес-логики
- Не кешируют альтернативные состояния

Search — это projection domain state, а не отдельная система правды.

### 8. Admin Is System Actor
Администратор — это системный актор, а не пользователь с расширенным UI. Любое admin-действие:
- Создаёт system event
- Распространяется через все домены
- Синхронизируется через realtime
- Имеет предсказуемый результат

Локальные admin-эффекты запрещены.

### 9. Realtime Marketplace Law
Realtime обновления не должны изменять бизнес-логику marketplace. SSE:
- Синхронизирует состояние
- Не вычисляет его
- Не создаёт новые состояния

Client всегда может восстановить truth после reconnect.

### 10. Marketplace Stability Principle
Добавление новых возможностей marketplace (promotions, dealer accounts, paid boosts, new vehicle types) не должно требовать изменения core модели listings.

Core marketplace model считается стабильной и расширяется только через дополнительные слои.

Приоритеты:
- Консистентность объявлений
- Предсказуемость видимости
- Доверие пользователей к данным
- Масштабируемость без изменения ядра системы
