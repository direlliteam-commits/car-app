# Observability Architecture

Система спроектирована так, чтобы её состояние, поведение и архитектурная целостность были наблюдаемы и объяснимы в любой момент времени.

Цель — обнаруживать проблемы ДО появления production-багов, рассинхронизаций или деградации marketplace.

## Принципы

### 1. System State Visibility
Любое состояние системы должно быть объяснимо через domain state. В любой момент должно быть понятно:
- Почему listing видим или скрыт
- Почему пользователь имеет или не имеет доступ
- Почему применён конкретный lifecycle state

Необъяснимые состояния считаются архитектурной ошибкой.

### 2. Event Traceability
Каждое изменение должно иметь причину. Любые изменения listings, subscriptions, permissions, visibility и ownership должны быть прослеживаемы через события системы:
- User action
- Admin action
- Lifecycle transition
- System rule

### 3. State Transition Observability
Все переходы lifecycle должны быть наблюдаемыми. Система должна уметь показать:
что изменилось → когда → почему → каким действием.

### 4. Realtime Health Awareness
Realtime слой должен позволять определить:
- Отстаёт ли клиент
- Потеряны ли события
- Произошёл ли ресинк
- Актуальна ли версия состояния

Realtime проблемы должны обнаруживаться системой автоматически.

### 5. Consistency Monitoring
Система должна уметь выявлять:
- Divergence между server и client state
- Рассинхронизацию фильтров и listings
- Нарушения SSOT
- Повторяющиеся состояния

Любая потенциальная рассогласованность должна фиксироваться как сигнал риска.

### 6. Permission Observability
Для каждого действия должно быть понятно: почему оно разрешено или запрещено.

Permissions должны быть объяснимыми, а не скрытыми внутри логики.

### 7. Admin Impact Tracking
Каждое admin-действие должно быть наблюдаемым как system-wide событие.

Система должна позволять определить: какие listings, пользователи и состояния были затронуты.

### 8. Marketplace Health Signals
Система должна отслеживать архитектурные сигналы:
- Рост сложных conditional logic
- Появление обходных решений
- Увеличение количества исключений
- Частые ресинхронизации realtime

Это рассматривается как ранний индикатор архитектурной деградации.

### 9. Failure Visibility
Ошибки не должны быть "тихими". Каждый failure должен:
- Иметь объяснимую причину
- Быть обнаруживаемым системой
- Позволять восстановление состояния

### 10. Debug Without Code Principle
Состояние marketplace должно быть объяснимо через domain события и состояния, без необходимости анализа исходного кода.

### 11. Predictive Stability Awareness
Система должна позволять обнаруживать зоны будущего риска:
- Перегруженные домены
- Слишком связанные модули
- Частые state transitions
- Нестабильные realtime участки

### 12. Platform Transparency Principle
Marketplace должен быть наблюдаем как живая система, а не как набор экранов UI.

Главный принцип: если поведение системы нельзя объяснить через наблюдаемое состояние — архитектура считается недостаточно прозрачной.
