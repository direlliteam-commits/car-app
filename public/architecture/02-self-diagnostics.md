# System Self-Diagnostics

Режим непрерывной диагностики — все изменения проходят эти проверки во время разработки.

Цель — раннее обнаружение архитектурных проблем ДО того, как они превращаются в технический долг, рассинхронизацию данных или production-баги.

## Обязательные проверки

### 1. Consistency Scan (постоянно)
После любых изменений автоматически проверяй:
- Существует ли более одного источника истины
- Появляются ли вычисляемые данные, сохранённые как отдельное состояние
- Может ли UI попасть в состояние, которого нет на сервере

Если да — это архитектурная проблема, требующая рефакторинга.

### 2. State Divergence Detection
Проверяй, может ли возникнуть расхождение между:
- Server state
- Centralized state
- Realtime updates (SSE)
- UI representation

Любая возможность divergence считается критической архитектурной ошибкой.

### 3. Hidden Coupling Detection
Выявляй скрытые зависимости между модулями:
- Когда изменение в одном домене неожиданно влияет на другой
- Когда компонент знает слишком много о чужой логике
- Когда бизнес-логика просачивается в UI

Такие зависимости должны устраняться через domain boundaries.

### 4. Admin Cascade Simulation
Перед финализацией изменений мысленно симулируй admin-действия:
- ban/unban
- listing moderation
- subscription change
- visibility change
- role change

Проверяй, что все system-wide последствия предсказуемы и консистентны.

### 5. Realtime Integrity Check
Каждое SSE событие должно:
- Быть идемпотентным
- Иметь versioning
- Не создавать альтернативное состояние
- Корректно восстанавливать синхронизацию после reconnect

Если событие может примениться дважды или нарушить порядок — архитектура неправильная.

### 6. Data Lifecycle Validation
Для каждой сущности проверяй полный lifecycle:
create → update → moderation → visibility → archive/delete.

Не допускаются состояния, которые нельзя объяснить бизнес-логикой.

### 7. Permission Leak Detection
Проверяй, существует ли путь, при котором:
- Пользователь видит данные без разрешения
- Действие возможно через обход UI
- Realtime событие раскрывает запрещённые данные

Server permissions — единственный источник безопасности.

### 8. Scalability Stress Thinking
Перед закреплением решения оцени, что произойдёт при:
- ×10 listings
- ×10 concurrent users
- ×10 realtime events
- Росте категорий и фильтров

Если решение масштабируется линейно только сейчас — оно требует переработки.

### 9. Architecture Smell Reporting
Если обнаруживается любой из признаков:
- Дублирование логики
- Conditional explosion
- Специальные исключения под отдельные кейсы
- Временные обходные решения

Остановись и сначала устрани архитектурную причину.

### 10. Continuous Architecture Awareness
Работай не как исполнитель задач, а как системный архитектор:
каждое изменение оценивается с точки зрения целостности всей платформы.

Главный приоритет: предсказуемость системы, консистентность данных, масштабируемость без переписывания.
