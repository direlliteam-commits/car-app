# Architecture Evolution Rules

Система находится в фазе архитектурного укрепления. Каждое изменение оценивается не только на корректность сейчас, но и на влияние на будущую эволюцию платформы.

Главная цель — предотвратить архитектурную деградацию при росте функциональности.

## Принципы

### 1. Feature Growth Rule
Новые функции не должны изменять существующее поведение системы. Добавление новой логики происходит через расширение домена, а не через модификацию уже стабильных потоков данных.

Если новая фича требует переписывания нескольких существующих модулей — архитектура нарушена.

### 2. No Special Cases Rule
Запрещено создавать отдельную логику под конкретные кейсы:
- Отдельные условия для отдельных типов listings
- Исключения для конкретных ролей
- Временные флаги поведения

Система должна решать задачи через обобщённую доменную модель.

### 3. Domain Boundary Protection
Каждый домен (Listings, Users, Vehicles, Subscriptions, Wallet, Admin, Realtime) должен оставаться изолированным.

Если модуль начинает напрямую зависеть от внутренней логики другого домена — требуется введение domain interface или события.

### 4. Model Expansion Rule
При добавлении новых типов транспорта, ролей пользователей или бизнес-логики: расширяется существующая модель, а не создаётся новая параллельная структура.

Единая модель listings обязательна для всех категорий.

### 5. Realtime Stability Rule
Рост realtime функциональности не должен увеличивать сложность состояния клиента.

SSE остаётся механизмом синхронизации, а не бизнес-логики.

### 6. Admin Predictability Rule
С увеличением возможностей админки система должна становиться более предсказуемой, а не более сложной.

Любое admin-действие:
- Детерминировано
- Обратимо (где возможно)
- Полностью отражается через system state

### 7. Data Shape Stability
Форма данных должна оставаться стабильной во времени. Нельзя менять структуру данных так, чтобы старые части системы начинали работать иначе.

Изменения происходят через versioning и расширение, а не ломающее изменение.

### 8. Complexity Budget
Перед добавлением новой логики оцени: увеличивает ли она когнитивную сложность системы.

Если новая функция делает понимание системы сложнее — требуется более простая архитектурная абстракция.

### 9. Future Feature Simulation
Перед финализацией решений мысленно учитывай будущие сценарии:
- Дилерские аккаунты
- Платные продвижения объявлений
- Дополнительные типы подписок
- Новые категории техники
- Международное расширение

Решение не должно блокировать эти направления.

### 10. Long-Term Integrity Principle
Предпочтение всегда отдаётся архитектурной чистоте, даже если решение требует большего рефакторинга сейчас.

Нельзя накапливать скрытый технический долг ради скорости.

Работай как архитектор долгоживущей платформы: каждое изменение должно упрощать будущие изменения, а не усложнять их.
